# Building in the Labs

The **Labs** section is where compositions get prototyped and stress-tested before they become public docs. A lab is a throwaway-or-graduate story that wires real `kai-*` elements together the way a consumer would — to prove a slot, a part, or a new pattern actually composes. The composer showcase, the custom message thread, and the chat-slot demos all started here.

This page covers the two things you'll do: **build** a lab, and **use** what you built.

## Build a lab

A lab is just a Storybook story titled under `Labs/…`. Add a `*.stories.tsx` in `src/elements/` (or extend an existing one):

```tsx
const meta = { title: 'Labs/My Composition' } satisfies Meta;
export default meta;

export const Demo = {
  render: () => {
    let el: HTMLElement | undefined;
    onMount(() => {
      if (!el) return;
      // 1. Array/object props are JS PROPERTIES, never attributes.
      (el as any).items = [/* … */];
      // 2. kai-* events are non-bubbling CustomEvents — listen on the element.
      el.addEventListener('kai-select', (e) => console.log((e as CustomEvent).detail));
    });
    return (
      <kai-prompt-input ref={(e) => (el = e as HTMLElement)}>
        <kai-menu slot="toolbar-start" trigger-icon="plus" />
      </kai-prompt-input>
    );
  },
};
```

Three things the shadow boundary forces (the same rules consumers hit):

- **Array/object data goes on as a property in `onMount`**, not as an attribute — an attribute is always a string. Reassign a *new* reference to re-render (don't mutate in place).
- **Events are non-bubbling `kai-*` CustomEvents** — `addEventListener` on the element itself; Solid's `onClick`/`on:` don't cross the shadow root.
- **Slots take light-DOM children** — `<x slot="toolbar-start">` projects into the element's named region.

For a fuller example, read `src/elements/composer-showcase.stories.tsx` (the composer lab) or `src/elements/labs-message-thread.stories.tsx`.

## Use what you built

Run Storybook and open your story under **Labs**:

```bash
npm run storybook   # http://localhost:6006
```

> **Note — restart for shadow CSS.** The `kai-*` elements inject `compiled.css` (built once at startup). A brand-new element, a changed component class, or a new `@theme` token/`@utility` will **not** hot-reload — kill and restart Storybook (`lsof -ti:6006 | xargs kill; npm run storybook`). Preview CSS *does* hot-reload, which is misleading. Story files (props, args, new variants) hot-reload fine.

Verify it both ways:

- **Visually** — toggle light/dark with the theme switch; a lab should read in both. Check the composition holds at narrow widths.
- **Behaviorally** — for anything event- or interaction-driven, add a Playwright IVP (independent visual/behavioral proof) rather than trusting the eye. The `tests/e2e/*-ivp.spec.ts` suites target lab/element stories by their story id (e.g. `labs-menu--plus-menu`) and assert real shadow-DOM behavior. Run them with the matching `npm run test:*-ivp` script.

## Graduating a lab

When a composition is proven, it gets **promoted into the public docs** (a guide or pattern at [ui.kitn.ai](https://ui.kitn.ai)) — the lab story stays here as the testing ground, the docs page becomes the consumer-facing version. Keep labs lean; they're for proving composition, not for documenting the consumer API (that's the docs site's job).
