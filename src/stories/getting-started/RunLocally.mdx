# Run the kit locally

Work from source when you want to change a component, add one, or just read the code with the dev server running. The repo is a single package, and this Storybook is its development surface.

## Set up

```bash
git clone https://github.com/kitn-ai/ui.git
cd ui
npm install
```

## Storybook is the dev loop

Every primitive, component, and element has stories. Storybook is where you build and eyeball them in isolation, light and dark.

```bash
npm run dev   # http://localhost:6006
```

`npm run dev` and `npm run storybook` are the same command — both build the CSS and launch Storybook on port 6006.

Stories live next to their source as `*.stories.tsx`, in three places that mirror the kit's layers:

- `src/ui/*.stories.tsx` — UI primitives (`Button`, `Switch`, `Resizable`).
- `src/components/*.stories.tsx` — feature components (`Message`, `Artifact`).
- `src/elements/*.stories.tsx` — the `kai-*` web components.

Editing a story or its source hot-reloads the open story. That's the tightest loop for visual work.

## Test

The suite runs on Vitest. Most behavior is tested at the Solid-component level with `@solidjs/testing-library`.

```bash
npm test            # full suite, once
npm run test:watch  # re-run on change while you work
npm run typecheck   # tsc across the Solid src, React wrappers, React tests, and the Node MCP
```

There's also a Storybook test project (`npm run test:storybook`) that runs against the stories in a real browser.

**Note:** Test the component, not the element. The custom-element facade isn't testable under jsdom — it needs a real browser. So unit-test the underlying Solid component (e.g. `Switch`), not `<kai-switch>`. Element-level behavior that needs a browser is covered by Playwright instead.

## Build

```bash
npm run build
```

This compiles the library bundles (the element bundle, the remote-card provider, the React wrappers, the barrel, the state layer, and the MCP), then — in the `postbuild` step — builds the theme tokens, regenerates the API metadata, and emits the schemas.

## Generated API metadata

`npm run build:api` (run for you by the build) reads the elements and emits the machine-readable specs that everything downstream consumes:

- `src/elements/element-meta.json` — every element's props, events, and defaults.
- `dist/custom-elements.json` — the custom elements manifest.
- The typed React wrappers and `llms.txt` / `llms-full.txt`.

These files drive the reference tables and playgrounds, the React adapter, and the for-AI-agents output. Treat them as generated: change the source, then regenerate — don't hand-edit them.
