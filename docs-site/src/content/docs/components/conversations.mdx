<Aside type="tip" title="When to use">
Reach for `<kc-conversations>` whenever you need a conversation switcher outside of a SolidJS context — inside a React, Vue, Svelte, or plain HTML app. Shadow DOM keeps the kit's styles isolated so the host page's CSS won't bleed in either direction. If you _are_ in SolidJS and need fine-grained layout control, compose the `ConversationList` primitive directly instead.
</Aside>

## Usage

Register the custom elements once, then set data as JS **properties** and listen for CustomEvents:

```html
<script type="module" src="https://cdn.jsdelivr.net/npm/@kitn.ai/chat/dist/kitn-chat.es.js"></script>

<kc-conversations id="sidebar" style="display:block; width:300px; height:100vh;"></kc-conversations>

<script type="module">
  const el = document.getElementById('sidebar');

  el.conversations = [
    {
      id: 'c-1', title: 'Web component architecture',
      scope: { type: 'collection' }, messageCount: 12,
      lastMessageAt: '2026-06-16T10:00:00.000Z',
      updatedAt: '2026-06-16T10:00:00.000Z',
    },
  ];
  el.activeId = 'c-1';

  el.addEventListener('kc-conversation-select', (e) => {
    el.activeId = e.detail.id; // keep the active highlight in sync
  });
  el.addEventListener('kc-new-chat', () => console.log('new chat'));
  el.addEventListener('kc-toggle-sidebar', () => console.log('sidebar toggled'));
</script>
```

In module-bundler projects, prefer the import path:

```js
import '@kitn.ai/chat/elements'; // registers all kc-* custom elements
```

**Providing data — two approaches:**

- **JS properties (recommended for dynamic data)** — set `el.conversations` or `el.groups` after mount. Both are non-scalar; they must be assigned as JS arrays, not HTML attributes. `conversations` is a flat list that the element auto-buckets by recency. `groups` is a pre-bucketed array — supply it when you want to control the section headers yourself (e.g. "Work", "Personal"). When `groups` is provided it takes precedence and `conversations` is ignored.

- **Declarative children (great for server-rendered markup)** — nest `<kc-conversation>` children directly in the tag. Each child carries its `id` as an attribute and its title as text content. No JS property wiring is needed; the element reads them on mount and watches for DOM changes via MutationObserver. Events fire identically to the property-driven path.

**Keeping the active row in sync** — set `activeId` (an attribute/property) to the id of the open conversation. Update it in your `kc-conversation-select` handler to keep the highlight current.

**Events** — the element fires three events; see the **Events** table below and the **Code** panel above for the exact per-framework listener syntax.

## Examples

Each example is live — expand **Code** to copy the exact snippet for your framework.

### Default list

A flat `conversations` array passed as a JS property — the element auto-buckets the items by recency. This is the most common starting point.

<Example client:visible tag="kc-conversations" />

### Active conversation highlighted

Set `activeId` to the id of the currently open conversation. The matching row is highlighted and the list scrolls it into view.

<Example client:visible tag="kc-conversations" config={{ activeId: 'c-1' }} data="withActive" />

### Pre-bucketed groups

Supply a `groups` property alongside `conversations` when you want to own the section headers. Each conversation's `groupId` links it to a group; ungrouped items fall through to an "Other" bucket.

<Example client:visible tag="kc-conversations" config={{ activeId: 'c-1' }} data="withGroups" />

### Empty state

When `conversations` is an empty array the element renders its built-in empty state, prompting the user to start a new chat.

<Example client:visible tag="kc-conversations" data="empty" />

### Declarative children

Nest `<kc-conversation>` elements directly in the tag — no JS property wiring needed. Each child carries its `id` as an attribute and its title as text content. The element reads them on mount and watches for DOM changes via MutationObserver, so this path works equally well in plain HTML, server-rendered markup, or any framework that sets innerHTML.

<Example client:visible tag="kc-conversations" data="declarative" />
