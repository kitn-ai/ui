<Aside type="tip" title="When to use">
Reach for `<kc-model-switcher>` when you're building your own chat header and want the model picker as a standalone piece — without pulling in the full `<kc-chat>` wrapper. It sits naturally beside a context meter or title in the header bar. The element stays invisible until at least two models are provided, so you can always pass the full list and let the component decide whether to show up.
</Aside>

## Usage

Register the custom elements once, then wire up `models` as a **JS property** and listen for `kc-model-change`:

```html
<script type="module">
  import '@kitn.ai/chat/elements';
</script>

<kc-model-switcher id="ms"></kc-model-switcher>

<script type="module">
  const ms = document.getElementById('ms');

  ms.models = [
    { id: 'claude-opus', name: 'Claude Opus', provider: 'Anthropic' },
    { id: 'claude-sonnet', name: 'Claude Sonnet', provider: 'Anthropic' },
  ];
  ms.currentModel = 'claude-opus';

  ms.addEventListener('kc-model-change', (e) => {
    ms.currentModel = e.detail.modelId; // reflect the selection back
  });
</script>
```

A few things worth knowing:

- **`models` is a JS property** — assign it with `el.models = [...]`. Each item is `{ id: string; name: string; provider?: string }`. Setting it as an HTML attribute won't work for arrays.
- **`currentModel`** is a plain attribute/property (the selected model's `id`). Omit it and the element defaults to the first model in the list.
- **The element only renders when two or more models are provided.** With zero or one model the trigger is invisible — safe to always pass the full list.
- **Reflect the selection.** After the user picks a model, set `el.currentModel = e.detail.modelId` so the trigger label updates to show the new selection.

**Declarative child API** — if you'd rather avoid a JS property assignment, you can compose `<kc-model>` light-DOM children instead:

```html
<kc-model-switcher id="ms">
  <kc-model id="gpt-4o" provider="OpenAI">GPT-4o</kc-model>
  <kc-model id="gpt-4o-mini" provider="OpenAI">GPT-4o mini</kc-model>
  <kc-model id="claude-sonnet" provider="Anthropic">Claude Sonnet</kc-model>
</kc-model-switcher>
```

Each `<kc-model>` carries `id` (required), `provider` (optional), and a text label as its text content. Children are light-DOM data carriers hidden by Shadow DOM — no JS property assignment needed. When you mix both approaches, `models` property items render first, declarative children after.

**Listening for events** — the element fires a single `kc-model-change` CustomEvent with `{ modelId: string }` whenever the user selects a model. See the **Events** table below for the full payload and the **Code** panel in the Preview above for the exact listener syntax in your framework.
