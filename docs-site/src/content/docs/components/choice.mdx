<Aside type="tip" title="When to use">
Reach for `<kc-choice>` whenever the model needs the user to pick exactly one option before proceeding — plan selection, template choice, tone preference, or any "next step" branch. It is a Card element: drop it standalone or let `<kc-cards>` dispatch it automatically from a stream of card envelopes. Prefer `<kc-confirm>` when you only need a yes/no, and `<kc-form>` when the user needs to fill multiple fields.
</Aside>

## Usage

Register the elements once, then set the `data` **JS property** (not an attribute — the value is an object) and listen for the `kc-card` CustomEvent:

```html
<script type="module" src="https://cdn.jsdelivr.net/npm/@kitn.ai/chat/dist/kitn-chat.es.js"></script>

<kc-choice id="ch" heading="Choose a plan"></kc-choice>

<script type="module">
  const el = document.getElementById('ch');

  // `data` is the CardEnvelope.data — always set as a JS property.
  el.data = {
    prompt: 'Which plan fits your team?',
    options: [
      { id: 'free',  label: 'Free',       meta: '$0' },
      { id: 'pro',   label: 'Pro',        meta: '$12/seat', recommended: true },
      { id: 'biz',   label: 'Business',   meta: '$29/seat' },
    ],
  };

  // All Card elements surface a single bubbling `kc-card` CustomEvent.
  el.addEventListener('kc-card', (e) => {
    const ev = e.detail; // CardEvent
    if (ev.kind === 'action') {
      console.log('User chose:', ev.action, ev.payload);
    }
  });
</script>
```

A few things worth knowing:

- **`data` is a JS property**, never an attribute. Assign it with `el.data = { … }` after the element upgrades.
- **`heading`** is a plain attribute (string), rendered in the card chrome above the radiogroup.
- **`cardId`** (attribute `card-id`) correlates every emitted `kc-card` event back to this card instance. Set it when you have a server-generated id; otherwise the element falls back to its DOM `id`.
- **Option fields** — the only required fields per option are `id` and `label`. Everything else is optional: `description` (secondary line), `meta` (trailing badge, e.g. a price), `recommended` (pill), `media` (leading image with `imageAlt`), `disabled` (inert + skipped in keyboard nav), and `payload` (opaque object echoed back in the event).
- **`allowOther`** adds a final selectable row that reveals an inline text input. Submit emits `action: '__other__'` with `payload: { text }`.
- **`submitLabel`** overrides the default "Submit" button label (set it on the `data` object, not as an attribute).
- **After submit** the card automatically renders a read-only summary. You can also force this state upfront by setting `el.resolution = { kind: 'action', action: 'option-id' }` — useful for restoring persisted choices on reload.
- **`kc-cards` / `renderCard`** dispatch `kc-choice` automatically when the card envelope carries `type: 'choice'` — no manual wiring needed in a streamed cards flow.

**Card-contract events** — `kc-choice` uses the shared card envelope, so it fires `kc-card` (not a dedicated event per action). The `detail` is a typed `CardEvent`: `{ kind: 'ready', cardId }` on mount, `{ kind: 'action', cardId, action, payload? }` on submit, and `{ kind: 'error', cardId, message }` for a malformed definition. There is no separate Events table for this element — inspect the **Code** panel in the Preview for the exact listener syntax per framework.

<Example client:visible tag="kc-choice" config={{ heading: 'New workspace', cardId: 'card-templates' }} data="templates" />

### Quick replies with free-text escape

`allowOther` appends an "Other…" row that reveals an inline text input when selected. The shared Submit stays disabled until the field is non-empty; submitting emits `action: '__other__'` with `payload: { text }`.

<Example client:visible tag="kc-choice" config={{ cardId: 'card-replies' }} data="quickReplies" />

### Resolved (read-only)

Set `el.resolution = { kind: 'action', action: 'option-id' }` after assigning `data` to render the post-submit read-only view — no radiogroup, no Submit button. Use this to restore a previously resolved card on page load.

<Example client:visible tag="kc-choice" config={{ heading: 'Choose a plan', cardId: 'card-resolved' }} data="resolved" />

### Error state

A malformed definition (empty `options` array) triggers the inline card error view and emits `{ kind: 'error' }` on `kc-card`. Wire up the listener to surface the message in your UI.

<Example client:visible tag="kc-choice" config={{ cardId: 'card-error' }} data="error" />
