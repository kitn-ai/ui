<Aside type="tip" title="When to use">
Reach for `<kc-form>` whenever an agent needs to collect structured input: onboarding questionnaires, task parameters, feedback forms, booking flows, or any agentic step that requires user data before continuing. The schema is the single source of truth for both the UI layout and validation, so you don't maintain them separately.
</Aside>

## Usage

Register the custom elements once, then assign the form definition as a **JS property** — not an HTML attribute — because it is a plain object:

```html
<script type="module" src="https://cdn.jsdelivr.net/npm/@kitn.ai/chat/dist/kitn-chat.es.js"></script>

<kc-form id="my-form"></kc-form>

<script type="module">
  import '@kitn.ai/chat/elements'; // registers all kc-* custom elements

  const form = document.querySelector('#my-form');

  // `data` is a JSON Schema object (type:'object') with optional x-kc-* UI hints.
  form.data = {
    type: 'object',
    title: 'How did we do?',
    required: ['rating'],
    'x-kc-submitLabel': 'Send feedback',
    properties: {
      rating: { type: 'integer', title: 'Overall rating', minimum: 1, maximum: 5, 'x-kc-widget': 'rating' },
      comments: { type: 'string', title: 'Comments', 'x-kc-widget': 'textarea' },
    },
  };

  // `kc-form` has no events of its own — it routes through the Card contract.
  // A valid submission arrives as a bubbling `kc-card` CustomEvent.
  form.addEventListener('kc-card', (e) => {
    const ev = e.detail; // CardEvent
    if (ev.kind === 'submit') console.log('submission data', ev.data);
    if (ev.kind === 'action') console.log('secondary action', ev.action);
  });
</script>
```

**Key things to know:**

- **`data` is a JS property** (not an attribute). Always assign it with `el.data = { … }`. Import `FormDefinition` from `@kitn.ai/chat` for full TypeScript typing.
- **The schema maps deterministically to widgets:** `string` → text input; `string` + `enum` → radio group or select; `string` + `format` → typed inputs (email, URL, date); `integer`/`number` → number input; `integer` + `x-kc-widget: 'slider'` → range slider; `integer` + `x-kc-widget: 'rating'` → star rating; `boolean` → switch (or `checkbox` via `x-kc-widget`); `array` → repeater, checkbox group, or tag list; nested `object` → fieldset.
- **`x-kc-*` UI hints live inside the schema** — `x-kc-order` controls field order, `x-kc-submitLabel` overrides the submit button copy, `x-kc-actions` adds secondary buttons (each fires an `action` CardEvent), `x-kc-placeholder` sets the field placeholder, and `x-kc-dismissible` adds a dismiss affordance.
- **Events use the Card contract** — `kc-form` never fires its own custom events. Instead it emits a single bubbling `kc-card` CustomEvent with a typed `CardEvent` in `detail`. The relevant kinds are: `submit` (valid submission; `detail.data` is the collected, coerced, validated object), `action` (secondary button clicked; `detail.action` is the id from `x-kc-actions`), `ready` (fired on mount), `dismiss` (when `x-kc-dismissible`), and `error` (malformed definition). See the **Code** panel in the Preview above for the full listener syntax per framework.
- **`heading`** is a scalar attribute that sets the card chrome title (falls back to `data.title` when omitted).
- **`resolution`** is a JS property. Set it to `{ kind: 'submit', data: { … } }` after a successful submission to swap the form for a read-only `<dl>` summary.
