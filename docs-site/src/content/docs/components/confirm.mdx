<Aside type="tip" title="When to use">
Reach for `<kc-confirm>` when the model proposes an action that the user should explicitly approve before it runs — deploying to production, deleting files, or sending an email. Tone `warning` signals caution, `danger` signals destructive intent with a visual icon; both use colour **and** iconography so the risk is never communicated by colour alone. For a larger set of equally-weighted choices that are not approval-gated, consider `<kc-choice>` instead.
</Aside>

## Usage

Register the custom elements once, then set the `data` **property** — it is a JS object, not an HTML attribute:

```html
<script type="module" src="https://cdn.jsdelivr.net/npm/@kitn.ai/chat/dist/kitn-chat.es.js"></script>

<kc-confirm heading="Run database migration?"></kc-confirm>

<script type="module">
  import '@kitn.ai/chat/elements'; // registers the custom elements

  const el = document.querySelector('kc-confirm');

  // `data` is a JS property — never an attribute.
  el.data = {
    body: 'This will apply 3 pending migrations to production. This cannot be undone.',
    tone: 'warning',
    actions: [
      { id: 'approve', label: 'Run migration', style: 'primary', default: true },
      { id: 'reject', label: 'Cancel' },
    ],
  };

  // Cards emit ONE `kc-card` CustomEvent carrying a typed CardEvent.
  el.addEventListener('kc-card', (e) => {
    const ev = e.detail; // { kind: 'action', cardId, action, payload? }
    if (ev.kind === 'action') console.log('chose', ev.action);
  });
</script>
```

A few things worth knowing:

- **`data` is a JS property.** Assign it with `el.data = { … }` — the shape is `ConfirmCardData` from `@kitn.ai/chat`. The element validates the payload on mount and renders an inline error if it is malformed (e.g. empty `actions`).
- **`heading`** is a plain attribute. It becomes the card chrome title — set it when you want a short, scannable header above the body.
- **`card-id`** is a plain attribute. It correlates every emitted `kc-card` event back to this card — essential when multiple cards appear in the same flow.
- **Action styles** — each action accepts `style: 'primary' | 'default' | 'destructive'`. Exactly one action can be `default: true`; it becomes the keyboard default and receives focus when `autofocus` is set (off by default so it does not steal focus mid-stream).
- **Tone** — `'warning'` adds a caution icon; `'danger'` adds a danger icon and uses destructive accent colouring alongside any `style: 'destructive'` action.
- **Resolving** — once any action is activated the card resolves itself (other actions disabled, the chosen one marked). To render the resolved state pre-hydrated, set the `resolution` **property**: `el.resolution = { kind: 'action', action: 'approve' }`.
- **Dismissible** — set `dismissible: true` in the data payload to show a close affordance; clicking it emits `{ kind: 'dismiss', cardId }` via `kc-card`.

## Examples

Each example is live — expand **Code** to copy the snippet for your framework.

### Approve or reject

The canonical two-action pattern: a `warning` tone with a primary confirm action and a plain cancel. This is the shape most tool-approval flows will use.

<Example client:visible tag="kc-confirm" config={{ heading: 'Run database migration?' }} />

### Destructive action

Use `tone: 'danger'` with `style: 'destructive'` on the confirm action for operations that permanently destroy data. Both the tone icon and the button colour signal the severity — never relying on colour alone.

<Example client:visible tag="kc-confirm" config={{ heading: 'Delete files?' }} data="destructive" />

### Choice set

More than two options works naturally — each action is a button. Omit the `heading` attribute when the data payload already includes a `heading` field.

<Example client:visible tag="kc-confirm" data="choiceSet" />

### Dismissible

Set `dismissible: true` in the data payload to add a close (×) affordance. Clicking it fires `{ kind: 'dismiss', cardId }` via the `kc-card` event — the card does not remove itself from the DOM.

<Example client:visible tag="kc-confirm" config={{ heading: 'Send email?' }} data="dismissible" />

### Resolved (read-only)

Set the `resolution` property to render the chromed read-only view — the chosen action is marked and all buttons are disabled. This is the state you restore when replaying a past conversation.

<Example client:visible tag="kc-confirm" config={{ heading: 'Run database migration?' }} data="resolved" />

### Error state

If the `data` payload is malformed (e.g. an empty `actions` array), the element renders its inline error state and fires `{ kind: 'error', cardId, message }` via `kc-card`. The host can surface a fallback or log the issue without crashing the surrounding UI.

<Example client:visible tag="kc-confirm" data="errorState" />
