Drop `<kc-chat>` into any page and have a working chat UI before your next coffee. This guide walks from install to a live, event-wired element in the fewest possible steps.

## Install

```bash
npm install @kitn.ai/chat
```

SolidJS users also need the peer dependency:

```bash
npm install solid-js
```

## Register the web components

One import registers every `kc-*` element in the browser's custom element registry. Call it once — at the top of your entry file or in a `<script type="module">` tag:

```js
import '@kitn.ai/chat/elements';
```

<Aside type="tip" title="No bundler? Use the CDN">
Load the element bundle directly — no install or build step required:

```html
<script type="module">
  import 'https://cdn.jsdelivr.net/npm/@kitn.ai/chat/dist/kitn-chat.es.js';
</script>
```

Pin an exact version in production (e.g. `@kitn.ai/chat@0.13.0/dist/...`) — the package is pre-1.0, so minor releases may include breaking changes.
</Aside>

## Your first chat

Set **rich data as JS properties** and listen for **CustomEvents**. This is the universal pattern across every framework — only the binding syntax changes.

```html
<kc-chat id="chat" style="display:block; height:100dvh;"></kc-chat>

<script type="module">
  import '@kitn.ai/chat/elements';

  const chat = document.getElementById('chat');

  // Seed an initial message
  chat.messages = [
    { id: '1', role: 'assistant', content: 'Hello! How can I help?' },
  ];

  // Handle user submissions
  chat.addEventListener('kc-submit', (e) => {
    const userMessage = { id: crypto.randomUUID(), role: 'user', content: e.detail.value };
    chat.messages = [...chat.messages, userMessage];
    // Call your model here, then append an assistant message
  });
</script>
```

Three things to know from this example:

- **`messages`** is a JS property (not an HTML attribute) — assign a new array each time to trigger a re-render.
- **`kc-submit`** fires when the user sends. `e.detail.value` holds the text; `e.detail.attachments` holds any staged files.
- Give the element an explicit height — it fills its block, scroll and all.

## Streaming a reply

`<kc-chat>` is transport-agnostic: it renders what you give it. To stream a response, add an empty assistant message first, then replace it with a new object on each chunk.

```js
chat.addEventListener('kc-submit', async (e) => {
  const userText = e.detail.value.trim();
  if (!userText) return;

  // 1. Show user message and an empty assistant placeholder
  const assistantId = crypto.randomUUID();
  chat.messages = [
    ...chat.messages,
    { id: crypto.randomUUID(), role: 'user', content: userText },
    { id: assistantId, role: 'assistant', content: '' },
  ];
  chat.loading = true;

  // 2. Stream from your endpoint (or a proxy to your model provider)
  const res = await fetch('/api/chat', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ message: userText }),
  });

  const reader = res.body.getReader();
  const decoder = new TextDecoder();
  let answer = '';

  while (true) {
    const { value, done } = await reader.read();
    if (done) break;
    answer += decoder.decode(value, { stream: true });

    // Replace with a NEW object — mutation in place won't re-render
    chat.messages = chat.messages.map((m) =>
      m.id === assistantId ? { ...m, content: answer } : m
    );
  }

  chat.loading = false;
});
```

## Usage by framework

The web component works identically in every framework. Only the syntax for binding properties and events differs.

<Aside type="note" title="Properties vs attributes">
Array and object data — `messages`, `models`, `context`, `attachments` — must be set as **JS properties**, not HTML attributes. Only scalar values (`placeholder`, `loading`, `theme`) work as attributes. This is the most common integration mistake.
</Aside>
